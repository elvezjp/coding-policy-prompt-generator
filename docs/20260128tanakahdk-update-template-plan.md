# Plan: promptテンプレート強化（20260128tanakahdk-update-template）

## 目的
- 当社ルールに沿った「共通システムプロンプト + 規約ごとのユーザープロンプト」構成を、ジェネレーターのデフォルトテンプレートとして安定運用できる状態にする。
- 曖昧な規約でも最低限の一貫性を保てるよう、システムプロンプト側に汎用の判断指針を組み込む。

## 現状の課題
- 生成テンプレートが規約文をそのまま貼るだけで、判断基準・対象範囲・例外が明示されない。
- 規約が曖昧な場合に、AIの判断がぶれやすい。
- システム/ユーザープロンプトの役割分担が構造化されていない。
- 判定の厳格度が固定されており、プロジェクトや状況に応じた調整ができない。

## 制約条件
- **Excel列の拡張は行わない**（入力データの構造は現状維持）
- 追加項目はテンプレートに固定値またはプレースホルダとして組み込む
- 柔軟性が必要な項目はCLIオプションで制御する
- 新規セクションの追記は「後続で手動補完する」運用を前提とする
- 手動補完の負荷を下げるため、未記載セクションには汎用的な「記入ガイド」や「例の雛形」を埋め込む

## 改善方針（今回の実装）

### 1. テンプレート構造の強化

#### システムプロンプト（共通部分）
| セクション | 内容 | 備考 |
|-----------|------|------|
| 役割 | AIオーディターとしての役割定義 | 現状維持 |
| 目的 | 規約準拠チェックのタスク定義 | 現状維持 |
| 出力形式 | JSON形式（rule_id, result, reason） | 現状維持 |
| 注意事項 | 判断指針を強化 | **拡張** |
| プロジェクト前提 | Java版、フレームワーク等 | **新規追加** |

#### 注意事項の強化内容
- 推論すべき項目（対象/判断基準/許容・禁止/例外/混在時の扱い）
- **判定の厳格度**（CLIオプションで切り替え）
  - `strict`: 疑わしい場合はNG（保守的）
  - `lenient`: 明らかな違反のみNG（寛容）
- **不完全情報の扱い**
  - コードが不完全な場合の判定方針
  - 情報が曖昧な場合の報告方法
- **判定不能時の扱い（strictnessと分離）**
  - 既定: 判定不能はNG（理由に不足情報を明記）
  - 将来: 厳格度とは別に `--indeterminate-policy` で切替可能にする余地を残す
- **用語の定義（運用指針）**
  - 「疑わしい」: 仕様解釈は可能だが違反かどうかが断定できない状態
  - 「判定不能」: 規約適用の前提情報が欠落し、判断の根拠が成立しない状態

#### ユーザープロンプト（規約ごと）
| セクション | 内容 | 備考 |
|-----------|------|------|
| 規約概要 | 規約ID、概要、分類、カテゴリ | 現状維持 |
| 重大度 | 必須/推奨、違反時の影響度 | **新規（プレースホルダ）** |
| 詳細ルール | 規約の具体的な説明 | 現状維持 |
| 適用範囲・例外 | 例外ケース、除外条件 | **新規（プレースホルダ）** |
| 準拠の具体例 | 良いコード例 | 現状維持（未記載） |
| 違反の具体例 | 悪いコード例 | 現状維持（未記載） |
| グレーゾーン例 | 判断が難しいケース | **新規（プレースホルダ）** |
| チェック対象コード | 分析対象コード | 現状維持 |

#### 未記載セクションの「記入ガイド」方針
- 「重大度」: `必須 / 推奨 / 任意` のいずれかを推奨し、影響度の一言を添える
- 「適用範囲・例外」: 対象（例: クラス/メソッド/変数）と除外条件（例: テストコードは除外）を最低限示す
- 「準拠の具体例 / 違反の具体例」: 2〜5行の最小コード例（言語は対象に合わせる）
- 「グレーゾーンの具体例」: 「見た目はNGだがOK」「見た目はOKだがNG」を1つずつ示す

#### 記入ガイドの具体的なテキスト案

```

**【準拠の具体例】**

```

**【違反の具体例】**

#### 記入ガイドの反映方針
- 既定テンプレートに「記入ガイド（記入例・雛形）」の文言を組み込む
- Jinjaテンプレート利用時は、必要に応じて同等のガイド文をテンプレート側で用意する

### 2. CLIオプションの追加

```
--strictness strict|lenient    判定の厳格度（デフォルト: strict）
--project-context TEXT         プロジェクト前提情報（例: "Java 17, Spring Boot 3.x"）
```

補足。
- `strictness` が想定外の値の場合はエラーとする
- `project-context` が未指定の場合はセクション自体を省略する

### 2.1 テンプレートで利用可能な変数

| 変数名 | 型 | デフォルト値 | 備考 |
|--------|---|-------------|------|
| `rule_id` | str | - | 必須（Excelから取得） |
| `summary` | str | - | 必須（Excelから取得） |
| `description` | str | `""` | Excelから取得、未記入時は空文字 |
| `classification` | str | `""` | Excelから取得、未記入時は空文字 |
| `category` | str | `""` | Excelから取得、未記入時は空文字 |
| `strictness` | str | `"strict"` | CLIオプションから取得 |
| `project_context` | str | `""` | CLIオプションから取得、未指定時は空文字（Jinja向け） |

### 3. 実装の工夫

#### PromptRendererの拡張
- `render()` メソッドに `strictness` と `project_context` パラメータを追加
- `BuiltinPromptRenderer` で厳格度に応じたテンプレート切り替え
- `JinjaPromptRenderer` にも同様のコンテキストを渡す（テンプレート変数として利用可能）
- 互換性維持のため、Jinjaテンプレートで未使用の変数があっても失敗しない設計とする
- 旧テンプレート互換の意味: 新変数を**参照しない限り**既存テンプレートは動作する
- Jinjaテンプレート向けに `project_context` は未指定時に空文字を渡し、`None` が出力されないようにする
- カスタムテンプレートでマーカー（`規約ID: <rule_id>` または旧形式 `【ルールID】\n<rule_id>`）が出力されない場合は警告し、再実行時の更新判定はできない

#### CLIの変更
- `cli.py` に新オプションを追加
- `generate_prompts()` 関数のシグネチャ拡張
- `project-context` の空文字はCLIで `None` に正規化する

#### ヘッダ解決の留意点
- 必須列の解決で曖昧一致が発生した場合はエラーとし、候補一覧を表示する
- 既存仕様の明文化であり、挙動変更は行わない

## テスト計画

### 正常系

| テスト項目 | 検証内容 |
|-----------|---------|
| strictモード出力 | 「疑わしい場合は違反」の文言が含まれる |
| lenientモード出力 | 「明らかな違反のみ」の文言が含まれる |
| project-context指定 | 指定した文字列がプロンプトに含まれる |
| project-context未指定 | プロジェクト前提セクションが省略される |
| 新セクションの存在 | 重大度、適用範囲、グレーゾーン例が含まれる |
| 既存テストの回帰 | 既存テストがすべてパスする |
| Jinjaテンプレート互換 | strictness, project_context が変数として利用可能 |
| Jinja旧テンプレート互換 | 新変数を使用しないテンプレートでもエラーにならない |

### 異常系・境界値

| テスト項目 | 検証内容 |
|-----------|---------|
| strictness不正値 | `--strictness invalid` でエラーメッセージが表示される |
| project-context空文字 | `--project-context ""` でセクションが省略される（Noneと同等） |
