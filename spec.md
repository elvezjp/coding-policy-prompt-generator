# coding-policy-prompt-generator仕様書

## 1. 概要

### 1.1 目的とスコープ

本ツールは、Excelのコーディング規約（1行=1ルール）を、AIオーディター向けのSystem Promptへ変換し、同一Excel内に「詳細シート」として展開するCLIである。

最低スコープ（MVP）は次のとおり。

- 一覧シートからルールを読み取る
- ルールごとに詳細シートを作成または更新する
- 一覧シートのリンク列に詳細シートへのHYPERLINKを設定する
- サンプルExcelで動作する

### 1.2 用語

- 一覧シート: ルールが1行ずつ並ぶシート
- 詳細シート: ルール単位のSystem Promptを出力するシート
- ルールID: 詳細シート名や出力JSONに使う識別子

## 2. 処理フロー

本ツールは以下の順序で処理を実行する。

```
┌─────────────────────────────────────────────────────────────┐
│  1. 入力ファイルの読み込み                                    │
│     └─ Excelファイル（.xlsx）を開く                          │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  2. 一覧シートの解決                                         │
│     ├─ --index-sheet で指定されたシートを選択                │
│     └─ 未指定時は先頭シートを使用                            │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  3. ヘッダ行の解析と列マッピング                             │
│     ├─ --header-row で指定された行からヘッダを読み取る       │
│     ├─ 必須列（ID、概要、リンク）を解決                      │
│     └─ オプション列（分類、カテゴリ、説明）を解決            │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  4. ルールの抽出（ヘッダ行以降を走査）                        │
│     ├─ 各行からルールID、概要、補足情報を取得                │
│     ├─ 空行や不完全な行はスキップ（警告を出力）              │
│     └─ 有効なルールのみ処理対象とする                        │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  5. ルールごとの処理（ループ）                               │
│     ├─ 5.1 詳細シート名の決定                                │
│     │      └─ PROMPT_<rule_id> 形式（31文字制限対応）        │
│     ├─ 5.2 詳細シートの作成または更新                        │
│     │      ├─ 既存シートがあればマーカーで判定して更新       │
│     │      └─ なければ新規作成                               │
│     ├─ 5.3 プロンプトの生成と書き込み                        │
│     │      ├─ テンプレートからSystem Promptを生成            │
│     │      └─ A1セルに書き込み（折り返し表示、列幅設定）     │
│     └─ 5.4 リンク列の更新                                    │
│            ├─ 既存ハイパーリンクをクリア                     │
│            ├─ HYPERLINK式を設定                              │
│            └─ リンクスタイル（青色・下線）を適用             │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│  6. 出力ファイルの保存                                       │
│     ├─ --dry-run 指定時は保存せず変更計画を表示              │
│     └─ 通常時は <stem>_with_prompts.xlsx に保存              │
└─────────────────────────────────────────────────────────────┘
```

### 2.1 dry-run時の動作

`--dry-run` 指定時は、ステップ6でファイル保存を行わず、以下の情報を標準出力に表示する。

- 処理対象ルール数
- 新規作成する詳細シートの一覧
- 更新する詳細シートの一覧
- スキップした行の一覧（理由とともに）

## 3. 入力仕様

### 3.1 対象ファイル

- 入力は `.xlsx` を前提とする
- マクロ（.xlsm）の実行は行わない

### 3.2 一覧シートの前提

- 1行=1ルール
- ヘッダ行を基準に列を解釈する

必要列（論理名）は次のとおり。

- ID列: ルールID
- 概要列: ルールの主文
- リンク列: 詳細シートへのリンクを書き込む列

補足。

- 本ツールは列の新規追加は行わない（既存の列にのみ書き込む）
- そのため、リンク列は既存のヘッダから解決できる必要がある
- 説明列は任意とし、リンク列と同一列になる場合は補足を扱わない
- ヘッダ名の比較はNFC正規化のうえで行う
- さらに、空白や区切り（例: スペース/アンダースコア/ハイフン/長音など）の揺れは吸収して解決を試みる

### 3.3 サンプルExcelへの適合要件

対象: `docs/ai-auditor-format/20260121AIオーディター形式サンプルコーディング規約.xlsx`

サンプルの特徴に合わせ、次を満たすこと。

- 一覧シート名 `コーディング規約一覧` を指定可能
- ヘッダ行番号 `3` を指定可能
- `説明` 列をリンク列として扱える（この場合、説明は参照しない）
- 先頭の説明行（1-2行目）を実質スキップできる

## 4. 出力仕様

### 4.1 出力ファイル

- 既定では入力と同じディレクトリに出力
- 既定の出力名は `<stem>_with_prompts.xlsx`
- `--output` で変更可能

### 4.2 一覧シート

リンク列に以下の書式でハイパーリンクを設定する。

- 一覧シートのリンク列にHYPERLINK式を設定する
- 表示名は既定で `詳細` とする
- 青色（#0000FF）・下線付きのフォントスタイルを適用する
- 既存のハイパーリンク（cell.hyperlink）はクリアしてから設定する

式の例（概念）。

- `=HYPERLINK("#'PROMPT_N-001'!A1","詳細")`

### 4.3 詳細シート

- 1ルールにつき1シート
- 既定の接頭辞は `PROMPT_`
- シート名は `PROMPT_<rule_id>` を基本とする

シート名の制約に対応すること。

- 31文字制限に収める
- `[]:*?/\\` を除去または置換する
- 重複時はサフィックスで解決する（例: `_2`）

#### 4.3.1 シートの内容

- A1セルからSystem Prompt本文を出力する
- 本文はテンプレートから生成する
- セルは折り返し表示（wrap_text）を有効にする
- A列の幅は80文字に設定する

最低限含める情報。

- ルールID
- ルール概要
- ルール補足（あれば）
- 出力形式（JSONの雛形）

## 5. 変換ロジック仕様

### 5.1 ルールの抽出

- ヘッダ行以降をデータ行として扱う
- ID列と概要列がともに空の行はスキップする
- ID列が空で概要のみある場合はスキップする（警告を出力）
- 概要列が空でIDのみある場合もスキップする（警告を出力）
- ID列と概要列の両方に値がある行のみ処理対象とする

### 5.2 詳細シート名の決定

- 既定: `PROMPT_` + ルールID
- ルールIDは文字列として扱う
- 正規化はシート名制約の範囲に限定する

## 6. CLI仕様

### 6.1 コマンド

想定エントリポイント。

- `coding-policy-prompt-generator`

想定実行例。

- `uv run coding-policy-prompt-generator input.xlsx`

### 6.2 オプション（MVPで優先）

入出力。

- `input.xlsx` (位置引数, 必須)
- `-o, --output` 出力パス
- `--dry-run` 変更内容の要約のみ表示（ファイルは出力しない）

シートとヘッダ。

- `--index-sheet` 一覧シート名（未指定時は先頭シートを使用）
- `--header-row` ヘッダ行番号（1始まり）

列マッピング。

- `--id-column` ID列名（既定: `項番`）
- `--summary-column` 概要列名（既定: `概要`）
- `--description-column` 説明列名（既定: `説明`）
- `--link-column` リンク列名（既定: 未指定時はヘッダ行の最右端の列を使用）

生成仕様。

- `--sheet-prefix` 詳細シート接頭辞（既定: `PROMPT_`）
- `--template` テンプレートファイルパス（Jinja2。未指定時は内蔵テンプレート）
- `--strictness` 判定の厳格度（`strict` or `lenient`。既定: `strict`）
- `--project-context` プロジェクト前提情報（未指定時は出力に含めない）

> 注: READMEに記載の `--output-format` はMVPスコープ外とし、将来仕様として検討する。

### 6.3 終了コード

- `0`: 正常終了（dry-run含む）
- `1`: エラー終了（シート未発見、必須列未解決、ファイル読み込み失敗など）

## 7. テンプレート仕様

### 7.1 概要

既定では内蔵テンプレートを使う。`--template` 指定時は外部テンプレートを使う。

テンプレートはJinja2形式とし、プロンプト生成の柔軟性と保守性を両立する。

生成されるプロンプトはマークダウン形式とする。これにより、構造化された可読性の高い出力を実現する。

### 7.2 テンプレートの構造

生成されるプロンプトは、役割の異なる2つの部分から構成される。

| 部分 | 役割 | 特徴 |
|-----|------|------|
| システムプロンプト | AIオーディターとしての共通指示 | 全ルールで共通 |
| ユーザープロンプト | 個別規約の詳細情報 | ルールごとに異なる |

#### 7.2.1 システムプロンプト

システムプロンプトは、AIオーディターの振る舞いを定義する共通部分である。

必須項目。

- 役割（AIオーディターとしての立場）
- 目的（規約準拠チェックのタスク）
- 出力形式（マークダウン形式の構造定義）
- 注意事項（判断指針、厳格度、判定不能時の扱い）

オプション項目。

- プロジェクト前提（言語、フレームワーク等。`--project-context` で指定された場合のみ出力）

システムプロンプトはルールに依存しないため、全ルールで同一の内容を出力する。ただし、CLIオプション（`--strictness`, `--project-context`）によって一部の内容が変化する。

#### 7.2.3 注意事項に含める内容

注意事項セクションには、AIオーディターの判定における共通の指針を含める。

##### 判定基準

`--strictness` オプションに応じた判定基準を含める。

共通事項。

- 判定は「OK」「NG」「要確認」の3段階で行う
- 重要度が高い問題を優先して報告する

`strict`モード（既定）の場合。

- 疑わしい場合は違反（NG）として判定し、理由に曖昧さを記載する
- 軽微な違反も見逃さない

`lenient`モードの場合。

- 明らかな違反のみNGとする
- 疑わしい場合はOKとし、理由に懸念点を記載する

用語の定義。

- 疑わしい: 仕様解釈は可能だが、違反かどうかが断定できない状態
- 判定不能: 規約適用の前提情報が欠落し、判断の根拠が成立しない状態

##### 情報不足時の対応

不完全なコードや情報不足の場合は、不足点を理由に記載したうえで、可能な範囲で判定する。

##### 判定不能時の扱い

- 判定不能な場合は `strictness` の値に関わらず、常にNGとして報告する
- 理由には判定不能となった原因（不足情報など）を記載する

##### 出力制約

- 出力は指定されたマークダウン形式のみとする
- 余計な前置きや後書きを付けない
- 違反がない場合も、サマリーと総評は必ず出力する

#### 7.2.2 ユーザープロンプト

ユーザープロンプトは、個別の規約情報を定義する部分である。

必須項目。

- 規約概要（規約ID、概要、分類、カテゴリ、重大度）
- 詳細ルール（規約の具体的な説明）
- 準拠の具体例（良いコード例）
- 違反の具体例（悪いコード例）
- チェック対象コードのプレースホルダ（分析対象の挿入位置）

オプション項目。

- 適用範囲・例外（対象と除外条件）
- グレーゾーンの具体例（判断が難しいケース）

規約概要の記入ガイド。

- 分類・カテゴリ: 判定精度向上のため、記入を推奨
- 重大度: 必要に応じて追加

ユーザープロンプトはExcelの各行から取得した値を埋め込むため、ルールごとに異なる内容となる。

### 7.3 内蔵テンプレートの管理

内蔵テンプレートは、ソースコード内にハードコードせず、外部ファイルとして管理する。

設計方針。

- テンプレートファイルはパッケージに同梱し、インストール時に利用可能とする
- システムプロンプトとユーザープロンプトは別ファイルとして分離する
- テンプレートの変更がロジックの変更と混在しないようにする
- 非エンジニアでもテンプレートの調整が可能な形式とする

分離の利点。

- テンプレートのみの変更がレビューしやすくなる
- 将来的に複数のテンプレートセットを切り替える機能を追加しやすくなる
- システムプロンプトのみ、またはユーザープロンプトのみのカスタマイズが可能になる

### 7.4 テンプレート変数

テンプレートに渡す値は辞書形式とする。

| 変数名 | 型 | 既定値 | 説明 |
|--------|---|--------|------|
| `rule_id` | str | - | ルールID（必須、Excelから取得） |
| `summary` | str | - | ルール概要（必須、Excelから取得） |
| `description` | str | `""` | ルール補足（Excelから取得、未記入時は空文字） |
| `classification` | str | `""` | 分類（Excelから取得、未記入時は空文字） |
| `category` | str | `""` | カテゴリ（Excelから取得、未記入時は空文字） |
| `strictness` | str | `"strict"` | 判定の厳格度（CLIオプションから取得） |
| `project_context` | str | `""` | プロジェクト前提情報（CLIオプションから取得、未指定時は空文字） |

補足。

- `category` と `classification` は一覧シートに該当列がある場合に自動的に取得する（列名の既定値: `カテゴリ`, `分類`）
- 該当列が存在しない場合は空文字を渡す（エラーにはしない）
- `project_context` は未指定時に空文字を渡す（`None` にはしない）

### 7.5 カスタムテンプレート

`--template` オプションで外部テンプレートを指定できる。

要件。

- Jinja2形式のテンプレートファイルを指定する
- テンプレートには7.4で定義した変数が渡される
- カスタムテンプレートでも、再実行時の更新判定のためマーカーを含めることを推奨する
  - 推奨マーカー形式: `規約ID: <rule_id>`
  - マーカーが含まれない場合は警告を出力し、再実行時の更新判定ができない旨を通知する

### 7.6 記入ガイド

内蔵テンプレートでは、未記載セクションに記入ガイド（記入例・雛形）を含める。

目的。

- 手動補完の負荷を下げる
- 記入すべき内容の指針を示す
- 一貫性のある規約定義を促進する

#### 7.6.1 必須項目の記入ガイド

必須項目のうち、Excelから値を取得できないセクションには、記入を推奨するガイドを含める。

対象セクション。

- 詳細ルール（規約の具体的な説明）
- 準拠の具体例（良いコード例）
- 違反の具体例（悪いコード例）

ガイドの方針。

- 「（未記載）」と表示し、記入例を提示する
- 規約の判定精度向上のため、記入を推奨する旨を示す

#### 7.6.2 オプション項目の記入ガイド

オプション項目には、必要に応じて追加できる旨のガイドを含める。

対象セクション。

- 適用範囲・例外（対象と除外条件）
- グレーゾーンの具体例（判断が難しいケース）

ガイドの方針。

- 「（未記載）」と表示し、記入例を提示する
- 必要に応じて追加してください、という任意性を示す

#### 7.6.3 規約概要の記入ガイド

規約概要セクション内の項目に対する記入ガイドを含める。

対象項目とガイドの方針。

| 項目 | ガイドの方針 |
|------|-------------|
| 分類 | 判定精度向上のため、記入を推奨 |
| カテゴリ | 判定精度向上のため、記入を推奨 |
| 重大度 | 必要に応じて追加 |

- 分類・カテゴリが未指定の場合のみ、推奨ガイドを表示する
- 重大度のガイドは常に表示する

## 8. エラーと検証

最低限の検証を行う。

- 対象シートが見つからない場合は明確に失敗
- 必須列（ID, 概要, リンク）が解決できない場合は失敗
- 失敗時は、解決できなかった対象名を表示する
- リンク列が未指定かつ最右端列の解決に失敗する場合も失敗とする
- 必須列の解決で曖昧一致が発生した場合は失敗とする
  - 例: `Ambiguous header 'ルールID' matched multiple columns: ルールID, ルール-ID`

## 9. 非機能要件

- 既存の一覧データを壊さないことを最優先にする
- 変更は「詳細シートの追加/更新」と「リンク列の書き込み」に限定する
- 可能な範囲で冪等にする（再実行で破壊的にならない）

### 9.1 再実行時の動作

再実行（同じ入力に対して再度ツールを実行）したとき、以下の動作とする。

- 既存の詳細シートが存在する場合でも、A1にルールIDのマーカーが含まれる場合のみ上書き更新する（シートの削除・再作成はしない）
  - マーカー形式: `規約ID: <rule_id>`（例: `規約ID: N-001`）
  - 後方互換: 旧形式 `【ルールID】\n<rule_id>` も更新対象とする
  - カスタムテンプレートでマーカーが出力されない場合、警告を出し、再実行時の更新判定はできない
- 一覧シートのリンク列は、既にHYPERLINKが入っていても再設定する
- 一覧シートのリンク列以外のセルは変更しない

## 10. テスト仕様

### 10.1 単体テスト

自動テストでは、以下の観点から検証を行う。

#### 基本機能

- ルール一覧からプロンプト付き詳細シートが正しく生成されること
- 生成されるHYPERLINK式が正しい形式であること
- シート名にアポストロフィ等の特殊文字が含まれる場合のエスケープ処理

#### 列解決

- ヘッダ名の揺れ（空白・区切り文字）を吸収した列マッピング
- 必須列が解決できない場合の明確なエラーメッセージ
- 曖昧一致が発生した場合のエラーハンドリング

#### シート名の安定性

- シート名の31文字制限・禁止文字への対応
- 重複シート名発生時のサフィックス付与（`_2` 等）による衝突回避
- 再実行時に同一シートが保持され、破壊的変更が発生しないこと

#### スキップ条件とNFC正規化

- ID列・概要列が空の行を正しくスキップすること
- ヘッダ名のNFC正規化による表記揺れ吸収
- 入力と出力が同一パスの場合の拒否

#### テンプレートとdry-run

- 外部テンプレート（Jinja2）の適用
- `--dry-run` 時にファイルが出力されず、変更計画が表示されること

### 10.2 手動での動作テスト

サンプルExcelを使用して、以下の手順で動作確認を行う。

#### 実行手順

1. サンプルファイルを準備する
   - `docs/ai-auditor-format/20260121AIオーディター形式サンプルコーディング規約.xlsx`

2. CLIを実行する
   ```bash
   uv run coding-policy-prompt-generator \
     docs/ai-auditor-format/20260121AIオーディター形式サンプルコーディング規約.xlsx \
     --index-sheet "コーディング規約一覧" \
     --header-row 3 \
     --link-column "説明"
   ```

3. 出力ファイルを確認する
   - `20260121AIオーディター形式サンプルコーディング規約_with_prompts.xlsx` が生成されること

#### チェック項目

- [ ] CLIが正常終了すること
- [ ] 一覧シートの説明列にリンクが設定されていること
- [ ] 説明列がリンクスタイル（青色・下線付き）で表示されていること
- [ ] リンクをクリックすると対応する詳細シートへ遷移すること
- [ ] 各ルールに対応する詳細シート（`PROMPT_<ルールID>`）が作成されていること
- [ ] 詳細シートのA1セルにプロンプト本文が出力されていること
- [ ] プロンプト本文が折り返し表示されていること
- [ ] A列の幅が適切に設定されていること
- [ ] 再実行しても一覧シートの既存データが破壊されないこと
- [ ] 再実行しても詳細シートが重複作成されないこと
